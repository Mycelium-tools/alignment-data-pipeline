# Design: holistic result provenance bundles (P1)

Status: spec, awaiting review · 2026-07-09 · branch `arda/dad-judge-rubric`

This is the **first** of two separate specs spun out of the viewer axes-editor
brainstorm. This one makes holistic Tag/Analyze results **self-describing and
comparable**. The second (the friendly axes editor, `evals/dad_axes.yaml` editing
in the viewer + an SDF filler section) is a separate doc and depends on nothing
here except that bundles exist first.

## Problem

Today the holistic lane writes **one** result slot per run:

- `<run>/audit/category_records.jsonl` — the extraction tag index
- `<run>/audit/holistic_dad_report.json` — the analyzer report

Re-running Tag/Analyze **overwrites** that slot. So the moment you edit
`evals/dad_axes.yaml`, switch the model (e.g. haiku → gemini), or change the
extraction prompt and re-tag, the previous results — and any record of *what
produced them* — are gone. There is no way to tell which axes/model/prompt a given
`category_records.jsonl` came from, and no way to compare two variants
side-by-side.

## Goal

Every Tag/Analyze produces a **bundle**: a self-describing directory holding the
results plus a snapshot of exactly what produced them (axes config, model, prompt,
git commit, timestamp). Bundles are keyed by a **fingerprint** of their expensive
inputs, so:

- clicking Tag with the *same* inputs resumes the matching bundle (no re-paying);
- editing the axes / switching the model / changing the extract prompt yields a
  *new* bundle — tags from two different schemas can never mix;
- two schema-or-model variants sit side-by-side for comparison.

Both the CLI (`evals/holistic_dad.py`) and the viewer write bundles the same way —
the logic lives in the pipeline layer, so CLI runs are not a provenance hole.

Out of scope (own spec): the axes-editor UI; the SDF axes lane; changing which
analyzers exist or what they compute.

## Storage layout

```
outputs/dad/runs/<run>/holistic/
  <YYYY-MM-DD_HH-MM>_<fp8>/          one bundle; <fp8> = first 8 hex of tag-fingerprint
    axes_snapshot.yaml              verbatim copy of the axes file used
    manifest.json                   provenance (below)
    category_records.jsonl          the tag index (the expensive, keyed artifact)
    report.json                     the analyzer report (cheap, regenerated by Analyze)
  latest -> <newest bundle>/        relative symlink, matches the outputs/*/latest idiom
```

`manifest.json`:

```json
{
  "tag_fingerprint": "<hex>",
  "created_at": "2026-07-09T14:30:00",
  "git_commit": "<short sha or null>",
  "model": "gemini-2.5-flash",            // resolved tag model (config default filled in, not null)
  "extract_prompt_sha": "<hex-or-null>",  // sha256 of the extract template text
  "records_tagged": 142,
  "analysis": {                            // updated each time Analyze runs in-bundle
    "config": { ... },                     // the analysis: params used
    "analyzers": ["distribution", "..."],
    "synth_model": "gemini-2.5-flash",
    "synth_prompt_sha": "<hex-or-null>",
    "analyzed_at": "2026-07-09T14:35:00"
  }
}
```

## Fingerprint — deliberately split

**Bundle identity = tag-fingerprint** = `sha256` over the canonical serialization of:

1. the **fields** (ordered; each field's name, kind, values, derived_from,
   required, target — i.e. what changes the tags), and
2. the **resolved model** string — the explicit `--model`/viewer override, else
   the config default resolved in its place (so a bare model never fingerprints as
   `""` while the effective model actually comes from `config.yaml`; a raw
   `tag_fingerprint(...)` call still canonicalizes an absent model to `""`), and
3. the **extract prompt** template text (or `""` if none).

These are exactly the inputs that determine the *tags*. Same fingerprint → same
tags are valid → resume into that bundle (skip already-tagged records via the
existing resume-safe logic). Any change → new fingerprint → new bundle.

**The `analysis:` block and the synthesis prompt are NOT in the identity.** They
only affect the *report*, which Analyze recomputes cheaply inside the existing
bundle (no re-tag). This avoids the trap where tweaking a quota or an analyzer
forces a costly re-tag when the tags didn't change.

**Known trade-off (accepted):** re-Analyze overwrites `report.json` within a
bundle, so two *analysis* configs over the *same* tags do not sit side-by-side —
only two *tag* variants do. Analysis is nearly free to regenerate, so this is
acceptable; `manifest.analysis` always records the config behind the current
report. Documented, not fixed.

Canonicalization: serialize the fields to a stable JSON string
(`json.dumps(..., sort_keys=True)` over a list of per-field dicts built from the
`Field` dataclass in registry order) so cosmetic YAML reordering/whitespace/comment
changes do **not** change the fingerprint — only semantic field changes do.

## Behavior

### Tag
1. Load fields (from `--axes` / the viewer's canonical file), model, extract prompt.
2. Compute the tag-fingerprint.
3. Scan `<run>/holistic/*/manifest.json` for a bundle with that fingerprint.
   - **Found** → resume into it: `index_path = <bundle>/category_records.jsonl`,
     existing records skipped (resume-safe). Refresh `latest` to point at it.
   - **Not found** → create `<run>/holistic/<ts>_<fp8>/`, write `axes_snapshot.yaml`
     and an initial `manifest.json`, set `index_path` there, point `latest` at it.
4. Tag as today into `index_path`. Update `manifest.records_tagged`.

### Analyze
Operates on a **selected** bundle (default = `latest`). Reads that bundle's
`category_records.jsonl`, runs analyzers + synthesis, writes `<bundle>/report.json`,
and updates `manifest.analysis`. Never re-tags.

### Bare-corpus input (no run dir)
`resolve_inputs` also accepts a bare `.jsonl`. Bundles live under a sibling
directory `<corpus-stem>.holistic/<ts>_<fp8>/` with the same contents. (Same rule,
different root.)

## Code changes

### `evals/holistic/pipeline.py`
- New `bundle.py` (or functions in `pipeline`) — pure, no streamlit:
  - `tag_fingerprint(fields, model, extract_template) -> str`
  - `resolve_bundle(inputs, fingerprint, *, create: bool) -> BundlePaths`
    returning the bundle dir + `index_path` + `report_path`; scans manifests for a
    match, creates + snapshots + writes manifest + repoints `latest` when creating.
  - `list_bundles(run_dir_or_holistic_root) -> list[BundleInfo]` (id, path,
    manifest) sorted newest-first.
- `Inputs.index_path` now points inside the resolved bundle. `resolve_inputs`
  gains an optional `bundle_id` (None = latest for reads; the fingerprint path is
  taken at Tag time via `resolve_bundle`). Keep the pass-through-`Inputs` contract.
- **Back-compat:** if a run has a legacy flat `audit/category_records.jsonl` and no
  `holistic/` dir, `list_bundles` surfaces it as an implicit read-only bundle
  `legacy` (manifest = `{}`), so old runs still render. New tags never write to the
  flat path.

### `evals/holistic_dad.py` (CLI)
- Compute the fingerprint from the loaded `fields` + `--model` + extract template.
- Route tag/report writes through `resolve_bundle`; write `axes_snapshot.yaml`
  (copy of `--axes`) and `manifest.json`. Add `--bundle` to pick a bundle for
  `--analyze-only`; default latest. `report_path_for` → the bundle's `report.json`.
- `git_commit`: reuse whatever `run_manifest` capture already uses
  (`shared/utils.py`), or `None` if unavailable.

### `viewer/loader.py`
- `list_bundles(run_dir)`; `category_records(run_dir, bundle_id=None)` and
  `holistic_report(run_dir, bundle_id=None)` read the chosen bundle (default
  latest), with the legacy flat fallback. `combined_index(run_dir, bundle_id=None)`
  threads the same argument so the facet index reflects the chosen bundle.

### Viewer pages
- `viewer/ui_pages/run_diversity.py`: a **bundle picker** (default latest) above
  the report; the three metrics + all report sections read the selected bundle.
  Tag/Analyze write through the pipeline bundle logic (Tag auto-resumes/creates;
  Analyze targets the selected bundle). Show the manifest (model, created_at,
  git_commit) beside the picker.
- Judge → "Score a run" (`judge_batch` / `evals/selection.py` facet source): same
  bundle picker, default latest, so facet filters come from a chosen bundle.

### In-page explanations (required, not polish)

The viewer already carries visible plain-language descriptions next to its
controls (the Tag/Analyze captions on the Run diversity page — verified rendering
live). Every bundle-facing control added by this spec gets the same treatment,
written as instructions for a person who has never read this spec:

- **Bundle picker** (both pages): a caption explaining what a bundle is — "each
  bundle is one tagging pass, keyed by its exact axes + model + prompt; pick one
  to view its tags and report; *latest* is the newest" — plus the selected
  bundle's manifest facts inline (model, created_at, records_tagged, git_commit).
- **Tag button caption**: updated to say the bundle behavior out loud — same
  axes/model/prompt resumes the matching bundle (already-tagged records skipped);
  any change to them starts a fresh bundle, so old results are never overwritten.
- **Analyze button caption**: updated to say it rewrites the *selected* bundle's
  report only (tags untouched, nearly free), and that the manifest records which
  analysis config produced the current report.
- **Legacy bundle**: when the implicit `legacy` bundle is shown, a caption says
  it's the pre-bundle flat index with no recorded provenance, and that the next
  Tag creates the first real bundle.

These captions are part of each page's acceptance: the feature is not done until
the explanations render next to the controls they describe.

## Testing (money paths first)

Offline, `stub_claude`, `tmp_path` + `PIPELINE_OUTPUT_ROOT`, ~seconds. New/changed:

- **Fingerprint stability:** identical (fields, model, prompt) → identical hash;
  reordering YAML comments/whitespace (same semantic fields) → identical hash;
  changing a field value / model / extract prompt → different hash.
- **Resume into matching bundle:** Tag, then Tag again with identical inputs →
  **zero** API calls, same bundle dir, records unchanged.
- **New bundle on changed inputs:** Tag, edit a field, Tag → two bundle dirs, each
  with its own `axes_snapshot.yaml`; tags never mixed.
- **Manifest + snapshot contents:** created on first Tag; `records_tagged` correct;
  `analysis` block written/updated on Analyze; `axes_snapshot.yaml` byte-equal to
  the input axes file.
- **`latest` pointer:** points at the newest-created bundle; Analyze on a
  non-latest selected bundle does not move it.
- **Back-compat:** a run with only the flat `audit/category_records.jsonl` renders
  via the implicit `legacy` bundle; loader reads it; Tag creates the first real
  bundle without touching the flat file.
- **Analyze targets selected bundle:** report written into the chosen bundle only.
- Malformed `manifest.json` in one bundle degrades that bundle (skipped/ warned),
  never crashes `list_bundles`.

Follow the existing style: drive through `pipeline.run` / `holistic_dad.main` /
`loader`, assert on files written and the `calls` list; derive constitution-shaped
expectations, assert timestamps/uuids by shape.

## Migration

No migration script. Legacy flat results are read in place via the implicit
`legacy` bundle; the first new Tag on a run creates `holistic/`. The old flat files
are left untouched (leave pre-existing artifacts alone).

## Open questions

- Bundle label: `<ts>_<fp8>` is machine-derived. Do we also want an optional
  human `--label` (like pipeline runs) folded into the dir name for readability?
  Default answer: no — the fingerprint is the identity; model + created_at in the
  manifest give readability. Revisit if bundle lists get hard to scan.
- Pruning old bundles: none for now (disk is cheap, corpora are small). A
  `--keep-latest N` prune could come later if needed.
